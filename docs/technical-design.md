# 战地1 图章自动生成工具 — 技术方案

## 1. 目标与约束

### 1.1 目标

用户传入一张图像，程序计算需要使用哪些图章、以何种方式摆放，使合成结果在视觉上尽可能接近（或可识别地概括）原图。

### 1.2 硬约束

| 约束 | 说明 |
|------|------|
| 图章种类 | **256** 种（完整图章库） |
| 图层上限 | 战地支持的可用图章层数 ≤ **40** |
| 变换 | 每层通常支持平移、旋转、缩放（是否非均匀缩放/镜像需与编辑器对齐） |
| 颜色 | 图章一般可染色；色板/透明度能力需实测核对 |
| 混合 | 默认按 painter’s algorithm 的 alpha 覆盖；若存在特殊混合模式需单独建模 |

### 1.3 核心认知

在「有限图元集合 + 仿射变换 + 40 层硬上限」下，**精确像素还原几乎不可能**。

正确目标应是：

> 在 ≤40 层约束下，从 **256** 种图章中选并做仿射组合，最小化视觉误差（或最大化可识别性）。

工作流程应强调**概括（abstraction）**，而非照片级重建：

```text
原图 → 语义/结构简化 → 形状分解 → 图章匹配与放置 → 层数压缩 → 微调
```

## 2. 问题建模

每个图章放置视为带 alpha 的精灵：

\[
S_i(x,y) = \mathrm{Stamp}_{t_i}\big(T_i^{-1}(x,y)\big) \cdot c_i
\]

| 参数 | 含义 |
|------|------|
| \(t_i\) | 图章类型（离散，**256** 种） |
| \(T_i\) | 2D 仿射：平移、旋转、缩放（及可能的镜像等） |
| \(c_i\) | 颜色（及可选透明度） |
| 层序 | 后画覆盖前画 |

最终合成：

\[
I_{\mathrm{pred}} = \mathrm{Composite}(S_1,\ldots,S_n),\quad n \le 40
\]

优化目标示例：

\[
\min\;\; \mathcal{L}(I_{\mathrm{pred}}, I_{\mathrm{target}}) + \lambda \cdot \text{复杂度}
\]

这是**混合整数 + 连续非线性**问题，组合空间极大，必须采用启发式 / 分层求解，不能暴力穷举。

### 2.1 规模含义：256 图章 × 40 层

层数预算仍是 40，但**字典宽度**从「约 30」变为 **256**，搜索与工程压力主要涨在「每一层候选类型」而非「能放多少层」：

| 维度 | 约 30 章时 | **256 章时** |
|------|------------|--------------|
| 单层类型选择 | 还可粗网格扫全库 | 全库 × 角度 × 尺度必须分级/剪枝，禁止每层朴素三重循环 |
| 预计算 | 可选 | **强烈建议**：多尺度 mask、SDF、方向直方图、FFT 模板等构建期缓存 |
| GPU | 加分项 | 批量代价评估几乎成为标配（尤其字典匹配阶段） |
| 图章冗余 | 较少 | 大量章几何相近（圆/环/盾/条的变体）→ 需要 **聚类 / 代表元 / 标签路由** |
| 用户心智 | 尚可枚举 | 必须按 `tags`（basic/shape/symbol/animal/…）做分级检索，不能 256 平铺硬搜 |

**策略原则**：先路由到小候选集（例如 5–20 个章），再在候选集上做位姿搜索；全库 256 只用于召回，不用于每一像素的穷举。

## 3. 关键技术点

### 3.1 图章资产与编辑器「数字孪生」

没有可靠的离线渲染器，后续算法无法验证。

#### 3.1.1 资产形态：以 SVG 为源（推荐）

**只有 SVG、没有现成 PNG 完全可行，且通常更好。**

| 对比 | SVG（矢量源） | 预渲染 PNG |
|------|----------------|------------|
| 任意分辨率 | 按需光栅化，边缘清晰 | 放大糊、缩小锯齿要多套 mip |
| 染色 | 可先改 fill 再渲染，或渲染白模后乘色 | 多依赖白模 × 颜色 |
| 几何匹配 | 可解析 path 做轮廓/SDF（进阶） | 只能从像素估轮廓 |
| 与游戏一致性 | 取决于 SVG 是否从官方资源准确描出 | 截图 PNG 可能已带 UI 缩放误差 |

**推荐管线：SVG 为唯一真源 → 运行时/构建时光栅化为 alpha 模板。**

```text
assets/stamps/*.svg
        │
        ▼
  规范化（viewBox、居中、去多余 transform）
        │
        ├─► 构建期：多尺度 PNG 缓存（可选，加速匹配）
        │
        └─► 运行时：cairosvg / resvg / skia 按目标尺寸 → RGBA
                    │
                    ▼
              白模/形状模 (alpha) + 可选原色
                    │
                    ▼
              合成器 / 匹配器使用
```

**SVG 图章规范（建议写入资产 README）：**

1. **viewBox 统一**：例如全部规范到 `0 0 100 100` 或保留原始比例但元数据记录 `anchor` 与 `native_size`
2. **旋转中心**：默认几何中心 = viewBox 中心；若与游戏枢轴不一致，在 `stamps.json` 里写 `pivot: [cx, cy]`
3. **颜色占位**：
   - 理想：形状使用可替换 fill（如 `#FFFFFF` 或 `currentColor`），表示「可染色模板」
   - 若 SVG 已是多色成品：区分「固定色装饰」与「主染色通道」，或光栅化后只用 alpha 作形状
4. **透明背景**：无不透明底矩形；最终 alpha 即造型
5. **单 path / 多 path**：允许复合路径；孔洞用 evenodd / path 布尔，光栅化后自然形成
6. **命名**：`stamp_id.svg` 与元数据 id 一致（如 `circle.svg`、`chevron.svg`）

**元数据示例 `assets/stamps/stamps.json`：**

```json
{
  "canvas": { "width": 320, "height": 320 },
  "stamps": [
    {
      "id": "circle",
      "svg": "circle.svg",
      "pivot": [0.5, 0.5],
      "tint_mode": "multiply_white",
      "min_scale": 0.05,
      "max_scale": 2.0,
      "tags": ["basic", "round"],
      "cluster": "disk_family"
    }
  ]
}
```

256 条目时务必保证：`id` 稳定、`tags`/`cluster` 可用于召回缩域；缺标签时可用离线脚本按形状描述子自动打簇。

**Python 侧光栅化选项：**

| 库 | 说明 |
|----|------|
| `cairosvg` + `cairocffi` | 常用；需 cairo 原生库 |
| `resvg` / `resvg-py` | 快、效果稳，推荐优先评估 |
| `skia-python` | 功能强，体积与环境较重 |
| `svglib` + `reportlab` | 纯路径有限，复杂 SVG 易翻车，不推荐作主路径 |
| Inkscape CLI | 批处理兜底，依赖外部程序 |

构建示例（概念）：

```python
# 白模：把 SVG 按 size 渲成 RGBA，RGB 置 255，保留 alpha 作为形状
rgba = rasterize_svg(svg_path, width=s, height=s)
mask = rgba[:, :, 3]
white_stamp = np.dstack([np.full_like(mask, 255)] * 3 + [mask])
```

染色：`out_rgb = tint_rgb * (mask/255)`（或按预乘 alpha 规则），再与画布 alpha-over。

**进阶（可选，非 MVP）：** 直接解析 SVG path → 多边形/SDF，匹配阶段走几何误差，精修仍可回到光栅域 loss。MVP 阶段「SVG → 像素模板」足够。

#### 3.1.2 数字孪生还需对齐的内容

1. **由 SVG 得到透明光栅模板**（见上；不必手持 PNG 资源包）
2. **对齐坐标系**：画布分辨率、原点、旋转中心（中心/角点）、缩放定义
3. **参数量化**：角度/缩放/位移在游戏中的步进
4. **颜色模型**：自由 RGB / 预置色板 / 是否可调透明度
5. **交付格式**：优先 JSON 摆放说明 + 预览图；存档注入/分享码属可选且需考虑合规

**验收标准**：给定同一层列表，离线合成预览与游戏内预览在几何与颜色上足够一致。

### 3.2 目标图像概括（预处理）

40 层预算极紧，必须先简化原图：

- 降采样 + 去噪（图章本身分辨率有限）
- 色彩量化（k-means / median cut，压到约 3–8 色）
- 边缘保持平滑（bilateral / guided filter）后再分割
- 显著性 / 主体抠图（优先还原脸、logo 轮廓；背景用 1–2 层大色块）
- 矢量/多边形近似（Potrace、RDP 折线简化）
- 分层表示：背景底色 → 主轮廓 → 内部色块 → 高光/细节

**经验法则**：先保证剪影可读，再上内部结构，最后才是眼睛/文字等细节。

### 3.3 形状分解

把图拆成「可用图章表达的零件」，可选路线（可组合）：

1. **区域分割**  
   SLIC / watershed / 颜色区域生长 → 每个 region 尝试用 1 个（或少量）图章覆盖。

2. **图章字典匹配（推荐核心，面向 256 库）**  
   对候选区域，**禁止**直接对 256 × 全角度 × 全尺度做朴素滑窗。应拆成：

   **（a）候选召回（短名单）**
   - 按区域几何特征路由：圆度、细长度、孔洞数、对称性、凸度等
   - 或对 256 个章的 mask 做形状描述子（Hu / 傅里叶描述子 / 多尺度 SDF 统计），与区域描述子近邻检索 top-k（k≈8–32）
   - 标签过滤：`tags` 含 basic/symbol/…，用户或分类器先缩域

   **（b）短名单内位姿匹配**
   - 归一化互相关（NCC）
   - 带 alpha 的 IoU / Dice
   - mask 的 Chamfer / 有向距离场（SDF）匹配
   - 角度/尺度用金字塔 + 粗步长，再局部细化

   **（c）库侧预处理（构建期一次）**
   - 全库多分辨率 mask / SDF 缓存
   - 近重复图章聚类（同一簇只留代表元参与召回，匹配阶段可回扩到簇内变体）

3. **骨架 / 笔画分解**  
   适合徽章、文字、纹章；细长结构用条状图章沿中心线摆放。

4. **CSG / 遮盖思维**  
   若只有覆盖没有挖空，孔洞用「底色 + 上层同色盖掉」等方式近似布尔差。

### 3.4 40 层约束下的放置优化

#### （1）贪婪残差匹配（MVP 首选）

```text
R = target
layers = []
while len(layers) < 40 and error(R) > ε:
    在图章库 × 位姿网格上找最能降低误差的放置
    合成进 canvas
    更新残差 R
```

类似 matching pursuit / 贪心字典编码。

#### （2）由粗到精

- 先在大尺度、低分辨率搜索大块
- 锁定后再局部连续精修位姿

#### （3）层数压缩 / 合并

- 同色相邻层合并
- 删除被完全遮挡的层
- 多块可被单个更大图章替代时做替换搜索

#### （4）连续位姿精修

对离散搜索初值，用可微渲染或数值梯度微调 \(tx, ty, \theta, s\)（及颜色）：

- Loss：α 加权 MSE / SSIM / 边缘损失 / 感知损失
- 优化器：Adam 等
- 图章类型 \(t_i\) 保持离散不变

#### （5）进阶：可微合成 + 离散类型搜索

- 连续参数：梯度下降
- 图章类型：Gumbel-softmax / 周期性重退火 / 进化算法  
  工程量大，作为 v2。

#### （6）搜索加速（256 库必做）

- **构建期**：256 章 × 多尺度 mask/SDF/描述子缓存；近重复聚类
- **运行期召回**：描述子 top-k 或规则路由，把每层有效字典从 256 压到 O(10)
- 短名单内再 FFT 相关 / 粗角度网格 / 金字塔
- Beam search 保留 top-k 部分解
- 残差已很低的区域早期剪枝
- GPU 上对「短名单 × 位姿粒子」批量打分，而不是 256 全量像素遍历

### 3.5 误差度量（偏人眼，而非纯 MSE）

照片纹理用纯 MSE 会把层浪费在噪点上。更合适：

- α 加权区域色差（CIELAB）
- 边缘图差异（Sobel/Canny 后 L1）
- SSIM / LPIPS（感知）
- 多目标权重：**轮廓 > 内部颜色 > 纹理**

对徽章/头像类输入，轮廓一致性往往比颜色绝对值更重要。

### 3.6 交付物分级

| 级别 | 内容 | 优先级 |
|------|------|--------|
| L1 | 预览图 + 摆放清单（类型、坐标、角度、缩放、颜色、顺序） | 必做 |
| L2 | 半自动辅助（overlay / 对照工具） | 可选 |
| L3 | 存档注入 / 网络 API | 低优；涉及逆向与 ToS，默认不做 |

## 5. 语言与 GPU 加速

### 5.1 语言选择

| 层级 | 推荐 | 原因 |
|------|------|------|
| 主语言 | **Python 3.12+**（uv 管理） | CV/ML 生态完整，迭代快 |
| 数据模型 | **Pydantic v2** | 与编辑器 JSON 一一对应 |
| 热点加速 | **PyTorch (CUDA)** 或 **CuPy** | 批量相关、可微渲染、精修 |
| 极致性能（可选） | C++/Rust + CUDA 扩展 | 仅当 Python 实测成瓶颈时 |
| GUI | Qt (PySide6) / Dear PyGui / Web (FastAPI + Canvas) | 预览、调参、对照图章库 |

**结论**：以 Python 为主工程语言即可；不建议一上来用纯 JS/TS 或纯 C# 作为核心算法语言。

### 5.2 GPU 适合加速的部分

1. **批量模板匹配（256 库下收益最大）**  
   短名单图章 × 角度 × 尺度在 ROI 上的相关/卷积；构建期也可 GPU 烘焙描述子。
2. **可微光栅化精修**  
   `grid_sample` 合成，对位姿与颜色反传；40 层在 GPU 上通常很快。
3. **距离变换 / 形态学 / 金字塔**  
   kornia、CuPy、OpenCV CUDA 等。

注：层数上限 40 不变，GPU 的核心价值是把「更宽的 256 字典」变成可交互时延，而不是堆更多层。

### 5.3 仍适合 CPU 的部分

- 图章类型离散决策、层序、启发式控制流
- 分割、矢量化（Potrace 等）
- I/O 与 GUI

**典型格局**：CPU 做控制流 + 离散搜索；GPU 做代价评估与连续精修。

### 5.4 推荐依赖（务实版）

```text
Python 3.11+
├── numpy / scipy
├── opencv-python
├── scikit-image
├── pillow
├── resvg / cairosvg       # SVG → RGBA（二选一，优先评估 resvg）
├── torch + torchvision    # CUDA 匹配与可微合成
├── kornia                 # 可选：可微几何/色彩
├── numba                  # 可选：小核 CPU JIT
└── PySide6 / Gradio       # 界面
```

无 NVIDIA GPU 时：CPU + 强力降采样/较粗离散网格仍可做 MVP；有 CUDA 后再迁移 matching 与 refine。

## 6. 分阶段落地

### Phase 0 — 资产与渲染器

- 齐套 **256** 图章 **SVG** + `stamps.json`（锚点、缩放范围、染色模式、tags/cluster）
- 实现 SVG → RGBA 光栅化（统一 viewBox/pivot）
- 实现 CPU 合成器（白模染色 + alpha-over），参数与游戏对齐
- 手工 JSON 层列表 → 预览图，验证坐标系
- **预烘焙**多尺度 mask（及可选 SDF/描述子）缓存；256 库下此项视为 Phase 0 范围而非远期优化
- （建议）离线近重复聚类，生成 `cluster` 字段

**出口**：离线渲染与游戏预览基本一致；任意 `stamp_id` 可稳定加载；缓存可一键重建。

### Phase 1 — MVP 贪婪版

- 输入图 → 色量化 + 主轮廓
- **先不用满 256**：用人工/自动筛出的小型「基础几何子集」（例如 8–20 个）跑通残差贪婪
- 输出 ≤40 层 + 预览

**出口**：结果「能认出是原图/原主体」；管线不依赖全库。

### Phase 2 — 全库 256 字典匹配

- 描述子召回 top-k → 短名单位姿匹配（禁止 256 全量朴素滑窗）
- 区域分割后局部拟合
- 遮挡感知的层序
- 簇内变体精修（代表元命中后再比同簇其他章）

### Phase 3 — GPU 精修与观感

- PyTorch 可微微调位姿/颜色
- SSIM / 边缘 loss
- 自动砍层、合并
- GUI：原图对照、逐层调试

## 7. 风险与常见坑

1. **旋转中心不一致** → 预览正确、游戏内全偏  
2. **alpha 预乘 vs 直通** → 边缘脏边、颜色发灰  
3. **层序错误** → 局部匹配分高但全局被遮挡  
4. **按照片纹理还原** → 层数瞬间耗尽；必须先扁平化、插画化  
5. **缩放越界** → 游戏最小/最大缩放限制导致无效层  
6. **镜像** → 镜像后可能等效「半个新字典项」，需纳入搜索  
7. **文字** → 通用图章极难还原；宜放弃、手工，或仅几何近似字重  
8. **ToS / 逆向** → 存档注入与在线接口需谨慎，默认以说明书+预览交付

## 9. 成功标准（产品向）

不宜只用 PSNR 作为唯一 KPI，建议同时关注：

1. **可识别性**：一定距离外能认出主体/logo（剪影优先）
2. **结构**：主色块数量与位置大致正确
3. **合法性**：≤40 层，游戏内效果与离线预览一致
4. **性能目标**（可调整）  
   - CPU MVP：单张约 1–2 分钟内  
   - GPU 版：单张约 10–20 秒更佳
