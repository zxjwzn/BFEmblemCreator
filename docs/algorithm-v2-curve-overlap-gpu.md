# 算法修订：曲线色块 + 超界重叠 + 线条质量 + GPU

> 状态：**已落地 v0.2 主路径**（`approx/blocks|curves|stamp_curves|match_curve|torch_render|line_quality|metrics|pipeline`）  
> 背景：v0.1 实现观感不理想、CPU 测试可达数分钟；需利用「图章可大幅放大并伸出画布、靠重叠造型」的真实编辑器能力。

---

## 0. 对用户方案的结论

**总体评价：值得作为主算法重写，比当前「小 ROI 像素盖章」更贴近战地徽章的制作方式。**

| 提案 | 判断 | 说明 |
|------|------|------|
| 图章可极大、可出画布，靠重叠造型 | **必须采用** | 编辑器本就如此；当前搜索把章限制在 ROI bbox 附近是错误先验 |
| 评分加入线条/不规则惩罚 | **必须采用** | 纯 IoU/色差会接受锯齿拼贴、碎边、多段折线假轮廓 |
| 原图 → 曲线拟合的重叠色块 → 用图章边缘曲线去贴 | **主路径推荐** | 把问题从「像素字典编码」升为「有限形状的布尔色块拼贴」 |
| 全流程 GPU | **拟合/打分必须 GPU**；I/O 与 JSON 可 CPU | 目标：单张 emoji 交互级（秒～十几秒），禁止分钟级网格穷举 |

当前失败模式（v0.1 实现）可归因于：

1. **尺度先验过小**：`width/height ≈ ROI`，不敢用「大圆只露一角」「大条横穿画布」等手法  
2. **目标是色量表格，不是可盖的块**：碎标签逼算法盖碎章  
3. **评分不罚丑**：边缘弯折、毛刺仍可能 IoU 尚可  
4. **CPU 上角度×尺度×召回三重循环**：5 分钟级不可接受

## 1. 问题的正确几何图像

徽章编辑器里，一层图章在画布上的可见部分是：

\[
V_i = \mathrm{Canvas} \cap T_i(\mathrm{Stamp}_{t_i})
\]

其中 \(T_i\) 允许 **很大的** 缩放与平移，使 \(T_i(\mathrm{Stamp})\) 的大部分在画布外。  
最终颜色由 **从底到顶的覆盖** 决定：上层不透明区域完全遮挡下层。

因此「拟合」不是 tile mosaic，而更接近：

> 用少量 **仿射变换后的模板形状** 做 **有序 CSG（并/盖）**，拼出目标色块边界。

这与「大圆形黄色脸 + 上层盖出嘴/眼」的手工做法一致。

## 2. 推荐主路径：Curve-Block Decomposition → Stamp Edge Matching

```text
原图
  │
  ▼
A. GPU 概括（少色、大块、光滑边界）     ── 不是碎像素标签
  │
  ▼
B. 色块曲线化（每块：外轮廓 + 孔洞，样条/折线简化）
  │
  ▼
C. 块装配顺序（底→顶：大面积、被依赖的块先放）
  │
  ▼
D. 对每个目标曲线段/闭曲线：
     在「图章边缘曲线库」里检索
     + 允许 大尺度 / 出画布 的仿射
     + 用重叠解释「只要画布内弧匹配」
  │
  ▼
E. GPU 可微或批量栅格化精修（位姿+是否采用）
  │
  ▼
F. 评分：相似度 × 线条质量 × 简洁度
  │
  ▼
EmblemDocument（left/top/width/height 可大幅越界）
```

### 2.1 与 v0.1 MP 的关系

| | v0.1 像素 MP | v0.2 曲线色块 |
|--|--------------|----------------|
| 目标 | 量化像素残差 | 光滑色块 + 边界曲线 |
| 图章用法 | 小章贴 ROI | **大章裁切 + 多层遮盖** |
| 匹配域 | mask IoU | **弧/轮廓 Chamfer + 法向** |
| 重叠 | 附带效果 | **一等公民**（上层挖形、下层透色） |
| 评分 | 色+边+α | **再加强线条规则度与层数惩罚** |
| 实现 | CPU 网格 | **GPU 批量** |

v0.1 代码可保留为渲染/JSON 基础设施；**搜索与评分按 v0.2 重写**。

## 3. 阶段设计

### A. 概括：为「大块重叠」服务

目标输出不是 200 个碎标签，而是 **≤ N_blocks（如 8–20）个光滑单色区域**（可重叠表示时用 layer 序）。

推荐：

1. 320 对齐 + α  
2. 强平滑 / 可选 GPU bilateral  
3. **少色** K=3–6（emoji 够用）  
4. **区域合并到大块**：禁止小碎岛（面积阈值提高）  
5. 每块做 **形态学闭运算 + 轮廓简化（RDP / 样条）**  
6. 输出 `ColorBlock{ color, contour_pts, signed_sdf, area, depth_hint }`

关键：边界必须 **光滑到像笔刷**，否则后续曲线匹配会去贴锯齿。

### B. 图章侧：边缘曲线库（构建一次，GPU 常驻）

对每个 stamp（至少 basic 子集，可扩 256）：

1. 高分辨率 mask → **最外轮廓**（及孔洞）  
2. 折线化 / 弧段分割（按曲率折点）  
3. 存：  
   - 归一化点序（弧长参数）  
   - 多尺度曲率描述子  
   - SDF 纹理（GPU）  
4. 预计算若干 **规范朝向**（主轴）

匹配不是「整章 IoU」，而是：

> 目标曲线段 \(c(s)\) 与 变换后图章轮廓 \(T(\gamma)\) 在 **画布窗口内** 的对齐。

出画布部分不进入损失（或只作软约束），这正是「大章只露一段弧」的数学表达。

### C. 重叠如何进入模型

显式允许三类操作：

| 操作 | 含义 | 例子 |
|------|------|------|
| **基底大章** | 极大 scale，中心可在画外 | 大 Circle 做脸 |
| **覆盖改色** | 上层同形或异形盖住部分 | 棕色盖嘴、白眼 |
| **异形裁切感** | 并不真挖洞，用上层底色/邻色盖出缺口 | 用 Square/Banner 切边 |

搜索时对每个新层估计：

- 它主要服务哪个 `ColorBlock`  
- 它与已放置层的 **画布内可见增量**  
- 增量边界是否更接近目标曲线  

### D. 位姿搜索（必须 GPU）

**禁止** Python 三重循环调 PIL。

推荐结构：

```text
对每个 (stamp_id, 候选弧段)：
  批量采样 M 个仿射粒子
    - scale_log 范围要大：如 width ∈ [0.2, 8] × canvas
    - translation 允许中心远在画外
    - angle 全周
  GPU：grid_sample stamp SDF/mask → 画布
  打分向量化
  top-k → 局部牛顿/Adam 精修
```

单层验收仍可用「真渲染器」投影到 JSON；**内环打分用可微/批量近似渲染**。

## 4. 评分体系（核心修订）

最终分不应只有相似度。建议：

\[
\mathrm{Score}
= S_{\mathrm{sim}}
\cdot S_{\mathrm{line}}
\cdot S_{\mathrm{simple}}
\]

或加性并带强惩罚：

\[
\mathrm{Score}
= w_s S_{\mathrm{sim}}
+ w_l S_{\mathrm{line}}
+ w_c S_{\mathrm{simple}}
- P_{\mathrm{ugly}}
\]

### 4.1 \(S_{\mathrm{sim}}\) 内容相似度（保留但降权）

- 主体 α-IoU  
- 加权 LAB 色差  
- 粗边缘一致  

### 4.2 \(S_{\mathrm{line}}\) 线条质量（新增，权重高）

对 **预测图可见边界**（颜色跳变或 α 边界）提取折线/像素链，计算：

| 项 | 低分情形 | 度量建议 |
|----|----------|----------|
| 折线密度 | 短距离内多次急转 | 单位弧长曲率变号次数、角点密度 |
| 锯齿 | 楼梯状、单像素毛刺 | 相对光滑样条的偏差；高频 Fourier 能量 |
| 碎边 | 大量短开放曲线 | 曲线段数量 / 总周长 |
| 平行毛刺 | 双线、重影描边 | 近距离平行线段对 |
| 与目标法向 | 边界方向乱 | 边界法向直方图差 |

**判定直觉**：人眼可接受「色略偏」，不能接受「边缘像碎纸」。  
故：若 `corner_density` 或 `jaggedness` 超阈，**直接 overall 封顶很低**（例如 ≤0.35），即使 IoU 高。

### 4.3 \(S_{\mathrm{simple}}\) 简洁度

- 层数惩罚：\(\exp(-\lambda n/40)\)  
- 有效层（可见像素过少）惩罚  
- 同色碎层合并奖励  

### 4.4 达标策略

emoji 验收示例：

- `S_sim ≥ 0.55`  
- `S_line ≥ 0.60`（硬门槛）  
- `overall ≥ 0.50`  
- 层数 ≤ 40  

**线条不达标 = 整体不达标**，避免再出现「分还不低但很难看」。

## 5. GPU 加速架构

### 5.1 技术选型

| 组件 | 建议 |
|------|------|
| 张量 | **PyTorch + CUDA** |
| 图章采样 | `grid_sample`（mask/SDF） |
| 形态学/边缘 | kornia 或手写 conv |
| 曲线距离 | 点集 Chamfer（GPU） |
| 真预览 JSON | 仍可用现有 `EmblemRenderer`（偶尔调用） |
| 无 CUDA 时 | 明确降级警告 + 极小搜索预算，不假装一样快 |

### 5.2 必须 GPU 的热点

1. 粒子仿射批量渲染打分  
2. SDF / Chamfer  
3. 线条特征（Sobel、曲率近似）  
4. 色量 k-means（可 GPU，非最大头）  

### 5.3 性能目标

| 场景 | 目标 |
|------|------|
| 构建 stamp 曲线库（256） | < 30s 一次，磁盘缓存 |
| 单张 emoji 拟合（GPU） | **≤ 15–30s** 可接受；争 **≤ 5–10s** |
| 单次打分 | ms～几十 ms 批量 |
| CI 无 GPU 测试 | 用微小合成图 + mock，不跑满层搜索 |

5 分钟 CPU 网格：**视为回归，测试中应 fail 或跳过**。

### 5.4 与现有 Render 的关系

- **外环**：`EmblemDocument` + `EmblemRenderer` 保证游戏一致  
- **内环**：`TorchStampRenderer`（可微/批量）只为搜索  
- 接受层前：可用快速 CPU/GPU 栅格对齐一次，避免 domain gap 过大

## 6. 大尺度 / 出画布的搜索先验（非常关键）

位姿采样建议对数尺度：

```text
log_scale ~ Uniform(log(0.15*L), log(6*L))   # L = 画布边长
cx ~ Uniform(-L, 2L)   # 中心可在画外
cy ~ Uniform(-L, 2L)
angle ~ Uniform(0, 360)
```

对「圆形类」stamp：优先大 scale + 中心靠近画布内目标圆心。  
对「长条类」：允许 length ≫ L，横穿画布。  
对「只贴一段弧」：优化的是 **可见弧 Chamfer**，不是整章 IoU。

整章 IoU 在「大章伸出画布」时会系统性偏低，**不能再当主分数**。

## 7. 修订后的模块划分（建议）

```text
approx/
  preprocess.py      # 大块光滑概括（可 GPU）
  curves.py          # 轮廓提取、RDP、弧段、曲率
  stamp_curves.py    # 图章曲线库 + 缓存
  match_curve.py     # GPU 弧匹配 / 大仿射粒子
  overlap.py         # 层序、可见增量、遮盖假设
  line_quality.py    # 线条不规则性评分
  metrics.py         # sim × line × simple
  torch_render.py    # GPU 批量 sprite
  pipeline.py        # 新主循环
```

CLI 保持 `bfemblem approx` / `score`；`score` 必须输出分项：`sim/line/simple/overall`。

## 8. 风险与对策

| 风险 | 对策 |
|------|------|
| 曲线匹配局部最优 | 多起点粒子 + 由粗到细（先 64² 再 320） |
| 可微渲染与 PIL 不一致 | 终局用真 Render 微调/重打分 |
| 重叠组合爆炸 | 按 ColorBlock 深度层序贪婪，不做全子集 |
| 复杂 stamp 边缘过碎 | 匹配前对 stamp 轮廓也做简化 |
| 无 GPU 用户 | 文档标明；CPU 仅 debug |

## 9. 是否采用「曲线色块」方案？

**采用，作为 v0.2 主方案。** 理由：

1. 与手工徽章语言一致（大形 + 遮盖）  
2. 自然支持出画布大尺度  
3. 线条质量可在「曲线域」定义，而不是事后看像素凑合  
4. GPU 批量仿射+SDF 是成熟路径，能消灭分钟级测试  

不采用的部分：

- 纯端到端黑盒网络（缺数据、难导出 JSON）  
- 继续加大 v0.1 网格密度（更慢、仍小尺度思维）
